## Summary

The meal planner page needs to go from placeholder to a fully functional recipe-to-grocery workflow. Users select recipes from their collection, the system extracts and aggregates ingredients into a grocery cart, and users can manage that cart (remove items, adjust quantities, add manual items). Plans are saved with full audit history, can be copied from previous plans, and serve as the foundation for weekly/monthly meal prep.

---

## Current State

### What Exists
- **`meal_plans` table** — schema in place with `meals JSONB`, `grocery_list JSONB`, `total_cost DECIMAL`, `status VARCHAR(20)`, `start_date`/`end_date`, `user_id`, timestamps
- **RLS policies** — user-scoped CRUD (own plans only)
- **`MealPlanCalendar.tsx`** — feature-complete weekly calendar UI with day/meal slot rendering, week navigation, summary stats. **Not integrated** into the planner page.
- **`MealPlanner.tsx`** — placeholder page ("under development" message + link to chat)
- **API hooks** — `useMealPlans()` and `useCreateMealPlan()` exist in `api.ts`
- **`recipes.ingredients JSONB`** — structured `{name, amount, unit, category, notes}` per ingredient, ready for extraction
- **`ingredients` reference table** — shared catalog with `common_names TEXT[]` for fuzzy matching (sparse, ~10 entries)

### What's Missing
- No recipe selection flow (the "pick list")
- No ingredient aggregation from selected recipes into a grocery cart
- No grocery cart UI (add, remove, adjust quantities, manual entries)
- No meal plan history view or copy-from-previous functionality
- No "send to meal plan" action from the Recipes page
- No edit/update flow for existing meal plans
- No audit trail (who created/edited, when)
- No delete/archive for meal plans
- `MealPlanCalendar.tsx` not wired to backend
- `createMealPlan` API only accepts `startDate`, `endDate`, `preferences` — no structured meal/grocery data

## Scope of Work

### Area 1: Schema & API Updates

#### Schema Changes

The existing `meal_plans` table structure is mostly adequate but needs refinements for audit tracking and structured grocery data.

**Add audit columns to `meal_plans`:**

```sql
ALTER TABLE meal_plans
  ADD COLUMN created_by UUID REFERENCES profiles(id),
  ADD COLUMN last_edited_by UUID REFERENCES profiles(id),
  ADD COLUMN copied_from UUID REFERENCES meal_plans(id),
  ADD COLUMN notes TEXT;
```

- `created_by` — the user who originally created the plan (distinct from `user_id` when household sharing arrives via MOP-0002)
- `last_edited_by` — last user to modify the plan, updated on every edit
- `copied_from` — references the source plan when created via "copy from previous"
- `notes` — freeform notes for the plan (e.g., "Thanksgiving week", "Low-carb trial")

**Backfill:** Set `created_by = user_id` for all existing rows.

#### Defined JSONB Structures

The `meals` and `grocery_list` JSONB columns need well-defined structures:

**`meals` JSONB structure:**

```typescript
// meals is keyed by ISO date string, then by meal slot
interface MealPlanMeals {
  [date: string]: {                    // "2026-03-16"
    breakfast?: PlannedMeal[];
    lunch?: PlannedMeal[];
    dinner?: PlannedMeal[];
    snacks?: PlannedMeal[];
  };
}

interface PlannedMeal {
  id: string;                          // unique ID for this meal slot entry
  recipeId: string;                    // FK to recipes.id
  recipeName: string;                  // denormalized for display without join
  servings: number;                    // can override recipe default
  prepTime?: number;                   // denormalized from recipe
  cookTime?: number;
  notes?: string;                      // e.g., "half portion for kids"
}
```

**`grocery_list` JSONB structure:**

```typescript
interface GroceryList {
  items: GroceryItem[];
  lastGenerated: string;               // ISO timestamp of last aggregation
}

interface GroceryItem {
  id: string;                          // unique ID for this line item
  name: string;                        // ingredient name (normalized)
  amount: number | null;               // aggregated quantity
  unit: string;                        // measurement unit
  category: string;                    // "produce", "dairy", "pantry", etc.
  sourceRecipes: string[];             // recipe IDs that contributed this ingredient
  isManual: boolean;                   // true if manually added (not from a recipe)
  isChecked: boolean;                  // checked off while shopping
  isRemoved: boolean;                  // soft-removed by user (still tracked for re-add)
  notes?: string;                      // e.g., "organic only", "brand preference"
}
```

### Area 3: Grocery Cart (Ingredient Aggregation)

The grocery cart is the core value of the meal planner — turning selected recipes into a shopping list.

#### Aggregation Logic

When recipes are added to a plan, aggregate all ingredients:

```
For each recipe in the meal plan:
  For each ingredient in recipe.ingredients:
    1. Normalize name: lowercase, trim, singularize
    2. Normalize unit: convert to canonical form (e.g., "tablespoons" → "tbsp")
    3. Scale amount by (plan servings / recipe default servings)
    4. Find matching existing item in grocery list:
       - Exact name match → combine amounts (if same unit)
       - Same ingredient, different units → convert if possible, else keep separate
    5. If no match → add new line item
    6. Track which recipe(s) contributed each item (sourceRecipes[])
```

**Unit consolidation examples:**
- 2 tbsp olive oil (Recipe A) + 3 tbsp olive oil (Recipe B) → 5 tbsp olive oil
- 1 cup flour (Recipe A) + 0.5 cups flour (Recipe B) → 1.5 cups flour
- 2 cloves garlic (Recipe A) + 3 cloves garlic (Recipe B) → 5 cloves garlic

**Edge cases:**
- Different units for same ingredient: keep separate rows if not convertible (e.g., "1 bunch cilantro" + "2 tbsp cilantro")
- Unitless amounts: "2 chicken breasts" + "4 chicken breasts" → "6 chicken breasts"
- Null amounts ("salt to taste"): show without quantity, don't aggregate

#### Grocery Cart UI

- **Grouped by category:** produce, dairy, protein, pantry, grains, condiments (using `ingredient.category`)
- **Per item:** name, aggregated amount + unit, source recipe count
- **Actions per item:**
  - Remove (soft delete — `isRemoved: true`, can undo)
  - Adjust quantity (manual override)
  - Edit notes (e.g., "organic", "Costco")
  - Check off (for shopping mode)
- **Manual add:** Button to add an ingredient not from any recipe (name, amount, unit, category)
- **Regenerate:** Button to re-aggregate from current recipe selection (resets manual edits with confirmation)
- **Shopping mode:** Simplified view for use in-store — large checkboxes, category headers, checked items move to bottom

#### Re-aggregation on Plan Changes

When recipes are added to or removed from the plan:
- Re-run aggregation on the full recipe set
- Preserve manual additions (`isManual: true` items survive re-aggregation)
- Preserve manual quantity overrides with a warning if the underlying recipe changed
- Preserve checked state

### Area 4: Meal Plan CRUD & History

#### Create Flow

1. User clicks "New Meal Plan" on the planner page
2. Picks date range (defaults to next Monday–Sunday)
3. Optionally: copy from a previous plan (see below)
4. Plan created with `status: 'draft'`, `created_by` set to current user
5. User adds recipes via picker modal or "Send to Meal Plan" from recipes page
6. Grocery cart auto-generates as recipes are added
7. User can finalize plan (set `status: 'active'`)

#### Edit Flow

- Any field on the plan is editable while in `draft` or `active` status
- Adding/removing recipes → re-aggregates grocery cart
- Adjusting servings → recalculates ingredient quantities
- `last_edited_by` and `updated_at` updated on every save

#### Status Lifecycle

```
draft → active → completed
         ↓
      archived
```

| Status | Meaning | Editable? |
|--------|---------|-----------|
| `draft` | Work in progress, not finalized | Yes |
| `active` | Current plan being followed | Yes |
| `completed` | Past plan, done | Read-only |
| `archived` | Soft-deleted / hidden from default views | Read-only |

```sql
-- Update status check constraint
ALTER TABLE meal_plans DROP CONSTRAINT IF EXISTS meal_plans_status_check;
ALTER TABLE meal_plans ADD CONSTRAINT meal_plans_status_check
  CHECK (status IN ('draft', 'active', 'completed', 'archived'));
```

#### Delete

- Soft delete via `status: 'archived'`
- Archived plans hidden from default list but accessible via "Show archived" toggle
- Hard delete available as a secondary action on archived plans

#### Copy from Previous Plan

1. User clicks "Copy from Previous" when creating a new plan
2. Shows list of past plans (sorted by date, most recent first)
3. Preview: shows which recipes were in the selected plan
4. Options:
   - Copy recipes + day/slot assignments (shift dates to new range)
   - Copy recipes only (unassigned, user re-assigns)
   - Copy grocery list as-is (with option to re-aggregate)
5. New plan created with `copied_from` set to source plan ID
6. All data is deep-copied — editing the new plan does not affect the source

#### History View

- Default planner view shows active/draft plans
- "History" tab or toggle shows completed + archived plans
- Each plan shows: title, date range, recipe count, created by, last edited by, timestamps
- Click to view read-only details of past plans
- "Copy" action available from history view

## Notes

- **Client-side aggregation.** Ingredient aggregation happens in the browser, not in an edge function. The recipe data is already loaded client-side and the logic is deterministic (no LLM needed). This avoids unnecessary API calls and keeps the interaction snappy.
- **JSONB storage, not relational.** Meals and grocery lists are stored as JSONB within `meal_plans` rather than in separate join tables. This matches the existing schema design and keeps plan data self-contained (a completed plan is a snapshot, not a live query). Tradeoff: harder to query individual meals across plans, but that's not a current use case.
- **Unit conversion already exists.** `RecipeDetail.tsx` already converts ingredient units between metric/imperial via `convertIngredient()` and `optimizeUnit()`. The aggregation utility should use these same functions for consistency.
- **`ingredients` reference table.** Currently too sparse (~10 entries) to be useful for ingredient normalization during aggregation. The aggregator should work primarily on string matching (lowercase + trim) rather than relying on the reference table. As the table grows (potentially via MOP-0003 allergen work), it can be incorporated.
- **Denormalized recipe data in `meals` JSONB.** Recipe name, prep time, cook time are stored inline so past plans remain readable even if the source recipe is edited or deleted. `recipeId` allows linking back to the current version when available.
- **MOP-0002 dependency (soft).** Meal plan sharing with household members is a P3 item here, matching MOP-0002's Area 4. The audit fields (`created_by`, `last_edited_by`) are designed to support multi-user households but work fine for single users in the meantime.
- **MOP-0003 integration.** When allergen detection ships, the grocery cart can show allergen warnings per item (e.g., "contains dairy" badge on parmesan). The `detected_allergens` data on recipes feeds into this naturally.
- **No AI-generated meal plans in v1.** The existing chat can suggest recipes, but automatic "generate a week of meals" is out of scope for this MOP. The focus is on manual recipe selection and grocery aggregation — the core workflow. AI-assisted plan generation can be a future MOP.

## Summary

Users and their dependents need dietary restriction and allergy profiles that actively inform the recipe experience. When a recipe is added or viewed, the system should identify ingredients that conflict with any household member's dietary needs and surface clear warnings. This requires: dietary/allergy fields on `profiles` for authenticated users, an ingredient-to-allergen mapping system, and UI integration on recipe pages.

---

## Current State

### What Exists
- **`family_members`** table already has `dietary_restrictions TEXT[]` and `allergies TEXT[]` — used for non-auth dependents (kids, etc.)
- **`recipes.dietary_tags TEXT[]`** — column exists in schema but is **completely unused** (never populated by extraction pipeline, never queried by frontend)
- **`recipes.ingredients JSONB`** — stores `{name, amount, unit, category, notes}` per ingredient; `category` is coarse ("dairy", "protein", "produce", "pantry", "grains", "condiments") but not allergen-aware
- **`ingredients` reference table** — has `name`, `category`, `subcategory`, `common_names TEXT[]` but only ~10 seed entries and no allergen data
- **Recipe extraction pipeline** — LLM already parses ingredients with categories; prompt defines 6 categories (protein, produce, pantry, dairy, grains, condiments)
- **`FamilyMembers.tsx`** — UI for setting dietary restrictions and allergies on dependents (badge picker + custom entries)

### What's Missing
- **No `dietary_restrictions` or `allergies` on `profiles`** — authenticated users cannot record their own dietary needs
- **No ingredient-to-allergen mapping** — no way to know that "parmesan" contains dairy or "soy sauce" contains soy + wheat
- **No allergen detection at recipe time** — recipes are saved without checking against household dietary profiles
- **No warnings on recipe display** — users see no indication that a recipe conflicts with their household's needs
- **`dietary_tags` never populated** — extraction pipeline doesn't classify recipes by allergen content
- **`ingredients` table too sparse** — 10 entries, no allergen metadata

### Area 2: Allergen Taxonomy & Ingredient Mapping

#### Allergen Categories

Use the FDA "Big 9" (FALCPA + FASTER Act) as the primary taxonomy, extended with EU-14 categories for broader coverage:

**Core 9 (FDA mandatory):**

| Category | Common Ingredient Examples |
|----------|---------------------------|
| Milk/Dairy | milk, cheese, butter, cream, yogurt, whey, casein, ghee, lactose |
| Eggs | eggs, egg whites, egg yolks, mayonnaise, meringue |
| Fish | salmon, tuna, cod, anchovies, fish sauce, Worcestershire sauce |
| Crustacean Shellfish | shrimp, crab, lobster, crawfish, prawn |
| Tree Nuts | almonds, walnuts, pecans, cashews, pistachios, pine nuts, macadamia |
| Peanuts | peanuts, peanut butter, peanut oil |
| Wheat | flour, bread, breadcrumbs, pasta, couscous, semolina, spelt |
| Soybeans | soy sauce, tofu, tempeh, edamame, miso, soybean oil |
| Sesame | sesame seeds, tahini, sesame oil, halvah |

**Extended (EU additional):**

| Category | Common Ingredient Examples |
|----------|---------------------------|
| Gluten | wheat (above) + rye, barley, oats (unless certified GF) |
| Celery | celery, celeriac, celery salt, celery seed |
| Mustard | mustard, mustard seeds, mustard powder |
| Lupin | lupin flour, lupin seeds |
| Molluscs | clams, mussels, oysters, squid, octopus, scallops |
| Sulphites | wine, dried fruits, vinegar (when > 10mg/kg) |

#### New Table: `allergen_ingredients`

A lookup table mapping ingredient terms to allergen categories:

```sql
CREATE TABLE allergen_ingredients (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  ingredient_term TEXT NOT NULL,           -- lowercase, e.g., "parmesan", "ghee", "tamari"
  allergen_category TEXT NOT NULL,         -- e.g., "dairy", "soy", "wheat"
  confidence TEXT NOT NULL DEFAULT 'definite'
    CHECK (confidence IN ('definite', 'probable', 'possible')),
  notes TEXT,                              -- e.g., "contains casein", "may be GF depending on brand"
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE (ingredient_term, allergen_category)
);

CREATE INDEX idx_allergen_ingredients_term ON allergen_ingredients (ingredient_term);
CREATE INDEX idx_allergen_ingredients_category ON allergen_ingredients (allergen_category);
```

**Confidence levels:**
- `definite` — always contains this allergen (e.g., cheese → dairy)
- `probable` — usually contains this allergen (e.g., Worcestershire sauce → fish)
- `possible` — may contain depending on brand/preparation (e.g., oats → gluten)

#### Seed Data Strategy

1. **Manual seed (~300-500 entries):** Curate the most common ingredient-to-allergen mappings. Reference the `food-allergens-ch` GitHub dataset and FDA guidance as starting points.
2. **LLM enrichment at extraction time:** During recipe extraction (Stage 2: Transform), use the existing LLM pipeline to classify each ingredient's allergen categories. The LLM already understands "parmesan → dairy" and "tamari → soy" — add a classification step to the transform stage prompt.
3. **Community growth:** As new recipes are added and new ingredient terms appear, unmatched terms are flagged for LLM classification and added to the lookup table.

#### Why Not an External API?

- **Spoonacular/Edamam** do ingredient-to-allergen mapping but are paid/rate-limited and add an external dependency
- **Open Food Facts** has allergen data but at the product level (barcodes), not ingredient level
- **USDA FoodData Central** has ingredient data but allergens aren't its focus
- The allergen categories are well-defined (9-14 categories) and the common ingredient mappings are a finite, manageable set (~500 covers the vast majority). Our existing LLM handles edge cases. No external API needed.

### Area 3: Allergen Detection in Recipe Pipeline

#### Detection Flow

When a recipe is saved (via extraction pipeline or manual entry):

```
ingredients[] → for each ingredient:
  1. Normalize: lowercase, trim, strip plurals
  2. Lookup: check allergen_ingredients table (exact match, then substring match)
  3. LLM fallback: if no match found, ask LLM to classify (and cache result back to table)
  4. Collect: aggregate all detected allergen categories
→ Store as recipe.detected_allergens TEXT[]
→ Compare against user + household dietary profiles
→ Generate warnings
```

#### Schema Change: `recipes.detected_allergens`

```sql
ALTER TABLE recipes
  ADD COLUMN detected_allergens TEXT[] DEFAULT '{}';
```

This stores the computed allergen categories for each recipe (e.g., `['dairy', 'wheat', 'eggs']`). Populated at save time, recomputed if ingredients change.

#### Populating `dietary_tags` (Existing Column)

The currently unused `dietary_tags` column can now be populated during the transform stage:
- If recipe contains no meat/fish → candidate for `vegetarian`
- If recipe contains no animal products → candidate for `vegan`
- If recipe contains no wheat/gluten → candidate for `gluten-free`
- If recipe contains no dairy → candidate for `dairy-free`
- Derived from `detected_allergens` + ingredient categories

#### Pipeline Changes (Transform Stage)

Add to `supabase/functions/recipe-pipeline/stages/transform.ts`:

```typescript
// After ingredient normalization:
// 1. Look up each ingredient.name in allergen_ingredients table
// 2. For unmatched ingredients, call LLM for classification
// 3. Cache new classifications back to allergen_ingredients
// 4. Set recipe.detected_allergens = unique allergen categories found
// 5. Derive recipe.dietary_tags from detected_allergens + ingredient categories
```

#### Backfill Existing Recipes

A one-time migration/script to process all existing recipes:
- For each recipe, run ingredients through the allergen detection flow
- Populate `detected_allergens` and `dietary_tags`
- Can be run as a Supabase Edge Function or a migration script

### Area 4: UI — Allergen Warnings on Recipes

#### Recipe Detail View

When viewing a recipe, if any `detected_allergens` conflict with the current user's profile or any household member's allergies:

- **Warning banner** at the top of the recipe: "This recipe contains **dairy** and **wheat**. [Member Name] has a dairy allergy."
- **Ingredient-level highlighting:** Flag specific ingredients that triggered the warning (e.g., red badge next to "parmesan" showing "dairy")
- **Severity indicators:** Use confidence levels — `definite` shows as a hard warning, `possible` shows as a softer caution

#### Recipe Card (List/Grid View)

- Small warning icon on recipe cards if any household conflicts detected
- Tooltip showing which allergens conflict and with whom

#### Recipe Save/Extraction (Chat)

When a recipe is extracted via chat and about to be saved:
- Show allergen detection results in `StructuredRecipeDisplay.tsx`
- If conflicts with household, show warning before save: "This recipe contains peanuts. [Child Name] has a peanut allergy. Save anyway?"
- User can still save — warnings are informational, not blocking

#### Recipe Search/Filtering

- Filter recipes by "safe for household" (no conflicts with any member)
- Filter by specific allergen exclusion ("no dairy recipes")
- Use `detected_allergens` column for efficient filtering

### Area 5: Household Dietary Aggregation

#### Query: All Household Dietary Needs

Once MOP-0002 households are in place and this MOP adds dietary data to `profiles`:

```sql
-- All allergies across the household (auth members + dependents)
SELECT DISTINCT unnest(allergies) AS allergy FROM (
  SELECT p.allergies FROM profiles p
  JOIN household_members hm ON hm.user_id = p.id
  WHERE hm.household_id = :household_id
  UNION ALL
  SELECT fm.allergies FROM family_members fm
  WHERE fm.household_id = :household_id AND fm.is_active = true
) combined;

-- Same pattern for dietary_restrictions
```

This powers features like:
- "Safe for everyone" recipe filter
- Allergen warning banners that name specific household members
- Future meal plan validation ("Monday's dinner contains peanuts but [Child] is allergic")

#### Dependency on MOP-0002

- Household aggregation requires `household_members` and `family_members.household_id` from MOP-0002
- Individual user dietary profiles and allergen detection can be built independently (no household dependency)
- Household-aware warnings are additive — system works for single users first, extends to households when MOP-0002 ships

## Notes

- **Allergen detection is informational, not medical.** The app should include a disclaimer that allergen detection is best-effort and users with severe allergies should always verify ingredients independently. This is a UX/legal requirement.
- **LLM is the escape hatch.** The lookup table handles the 90% case; the LLM handles ambiguous ingredients ("Worcestershire sauce" → fish, "marshmallows" → possible egg). Since we already have the LLM pipeline via OpenRouter, this adds minimal cost.
- **No external API dependency.** The 9-14 allergen categories are well-defined and the common ingredient set is finite (~500 entries covers the vast majority). Paid APIs like Spoonacular/Edamam add cost and latency for a problem we can solve locally.
- **`detected_allergens` vs `dietary_tags`:** These serve different purposes. `detected_allergens` is a factual list of allergen categories present in the recipe (dairy, eggs, wheat). `dietary_tags` is a derived classification (vegetarian, gluten-free, dairy-free) — the inverse/complement of allergens plus ingredient category analysis.
- **Confidence levels are P3.** Initial implementation treats all matches as definite. The `confidence` column in `allergen_ingredients` exists for future nuance (e.g., "oats may contain gluten depending on processing") but the UI can show all warnings equally in v1.
- **Dependency on MOP-0002 is soft.** Individual dietary profiles and per-recipe allergen detection work without households. Household-wide aggregation and "this affects [Member Name]" warnings are additive features that layer on once MOP-0002 ships.
- **Backfill strategy:** Run allergen detection on all existing recipes as a batch job. Can be a one-time Edge Function invocation or a migration script. LLM calls for unmatched ingredients should be rate-limited to avoid cost spikes.

## Summary

The project currently has zero automated tests and no testing framework installed. All verification has been manual. This MOP introduces a testing stack (Vitest + Testing Library + Playwright), establishes coverage targets, and prioritizes test authoring by risk — starting with the data layer (RPC functions, API client, RLS policies) and working up to UI components and end-to-end flows.

---

## Current State

- **No test runner** — `package.json` has no test-related dependencies (no Vitest, Jest, Playwright, or Cypress)
- **No test files** — zero `*.test.ts` or `*.spec.ts` files in `src/`
- **No CI test step** — build pipeline runs `tsc && vite build` only
- **Manual testing** — all features verified by hand (signup, invite flow, recipe CRUD, visibility, reactions, role management)
- **Vite-based project** — Vitest is the natural test runner (shares Vite config, fast HMR)

## Scope of Work

### Phase 0: Test Infrastructure Setup
**Files affected:** `package.json`, `vite.config.ts`, `src/test/setup.ts` (new), `tsconfig.json`

1. Install Vitest, `@testing-library/react`, `@testing-library/jest-dom`, `@testing-library/user-event`, `jsdom`, `msw` (Mock Service Worker)
2. Add Vitest config to `vite.config.ts` (jsdom environment, setup file, coverage thresholds)
3. Create `src/test/setup.ts` with Testing Library matchers and global mocks (Supabase client, auth)
4. Add `npm run test` and `npm run test:coverage` scripts
5. Add `src/test/mocks/supabase.ts` — mock Supabase client factory for unit tests
6. Update `tsconfig.json` with Vitest type references

### Phase 1: Data Layer Unit Tests (highest risk)
**Files affected:** `src/services/__tests__/api.test.ts` (new), `src/stores/__tests__/authStore.test.ts` (new)

These are the highest-value tests because the data layer mediates all Supabase interactions and bugs here cause data leakage or corruption.

| Test suite | What it covers | Key scenarios |
|-----------|---------------|---------------|
| `api.test.ts` — Household | `getMyHousehold`, `updateMemberRole`, `removeHouseholdMember`, `transferOwnership` | RPC call params, error propagation, `snakeToCamel` mapping |
| `api.test.ts` — Recipes | `getHouseholdRecipes`, `getPublicRecipes`, `getRecipeReactions`, `toggleRecipeReaction` | Author profile flattening, empty arrays, race condition (toggle is atomic RPC) |
| `api.test.ts` — Invites | `createHouseholdInvite`, `getMyPendingInvites`, `respondToInvite` | Accept → join household, decline → no join, edge function URL construction |
| `api.test.ts` — Collections | CRUD + `addRecipeToCollection`, `removeRecipeFromCollection` | Create, rename, delete, recipe add/remove |
| `api.test.ts` — Utility | `snakeToCamel`, `camelToSnake` | Nested objects, arrays, null handling, already-camel input |
| `authStore.test.ts` | Auth initialization, household loading, sign-out cleanup | Session restore, household membership set on login |

### Phase 2: RLS & RPC Integration Tests (requires local Supabase)
**Files affected:** `supabase/tests/` (new directory), `supabase/tests/rls.test.sql` or `supabase/tests/rpc.test.ts`

These tests run against a real local Supabase instance (`supabase db reset` + seed) to verify that RLS policies and RPC functions behave correctly with actual PostgreSQL.

| Test | What it verifies |
|------|-----------------|
| Household recipe visibility | User A (household) sees User B's `household` recipes; User C (outsider) does not |
| Private recipe isolation | No user can see another user's `private` recipes |
| Public recipe access | All authenticated users see `public` recipes |
| Role-gated operations | Only owner can update member roles; member cannot promote self |
| Member removal | Owner can remove member; member can leave (self-delete); member cannot remove others |
| Invite flow | Pending invite visible to invitee; accepted invite adds to household; duplicate invite blocked |
| RPC `get_my_household` | Returns correct members, dependents, pending invites for calling user |
| RPC `toggle_recipe_reaction` | Add, toggle, remove reactions atomically |
| Dependent management | `managed_by` set correctly; only household members can create dependents |
| Function search paths | All `SECURITY DEFINER` functions have `search_path = public` set |

**Approach options:**
- **pgTAP** — PostgreSQL native test framework, runs as SQL. Good for pure RLS/RPC testing.
- **Vitest + Supabase client** — TypeScript tests that connect to local Supabase. Easier to write, tests the full stack from client to DB.
- **Hybrid** — pgTAP for RLS policy matrix, Vitest for RPC return shapes.

### Phase 3: Component Tests
**Files affected:** `src/pages/__tests__/Household.test.tsx` (new), `src/components/__tests__/RecipeCard.test.tsx` (new), etc.

Test interactive UI behavior with mocked API responses (MSW).

| Component | Key scenarios |
|-----------|--------------|
| `Household` | Invite form submit, member role dropdown, remove confirmation dialog, dependent CRUD form |
| `RecipeCard` | Reaction toggle, visibility badge, author attribution |
| `VisibilityPicker` | Selection changes, correct icons per visibility level |

### Phase 4: End-to-End Tests (Playwright)
**Files affected:** `e2e/` (new directory), `playwright.config.ts` (new)

Full browser tests against a running dev environment.

| Flow | Steps |
|------|-------|
| Signup → household created | Register → complete setup → verify household exists |
| Invite → accept → shared recipes | Owner invites email → invitee signs up → accepts → sees household recipes |
| Recipe lifecycle | Create recipe → set visibility → verify in correct feed (private/household/public) |
| Role management | Owner promotes member → member sees admin UI → owner demotes → admin UI gone |

## Notes

- **Vitest over Jest**: Vitest shares the Vite transform pipeline, so TypeScript path aliases (`@/`), JSX, and CSS modules work without extra config. It's also significantly faster for Vite projects.
- **MSW for mocking**: Mock Service Worker intercepts at the network level, so tests exercise the real Supabase client code (including `.from()`, `.rpc()`, `.select()` chains) rather than mocking Supabase internals. This catches more bugs than mocking the client directly.
- **Local Supabase for RLS tests**: RLS policies can only be properly tested against a real PostgreSQL instance. `supabase start` + `supabase db reset` provides a clean local environment. Tests should use `supabase.auth.admin` to create test users with known IDs.
- **pgTAP consideration**: For pure SQL-level RLS testing, pgTAP is more thorough (can `SET ROLE` to simulate different users). However, it requires installing the pgTAP extension and writing SQL test files. Evaluate during Phase 2 implementation.
- **Coverage targets**: Start pragmatic — don't aim for 80%+ overall coverage immediately. Focus on the data layer where bugs have the highest blast radius. UI coverage can grow organically.
- **Edge function testing**: Supabase Edge Functions (Deno) require separate test tooling (`deno test`). Consider a Phase 5 for edge function unit tests if the scope grows.

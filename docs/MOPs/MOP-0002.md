## Summary

Users currently own recipes in complete isolation — there's no way to share recipes with family members, a household, or the public in a controlled way. This MOP introduces a three-tier permission model (private / household / public), household management with invite flows, and recipe collections (folders) with inherited sharing permissions.

---

## Current State

### What Exists
- **`profiles.family_id`** (UUID) — groups users into a family, auto-generated on signup (each user gets their own unique `family_id`)
- **`family_members`** table — stores non-authenticated family members (name, relationship, dietary restrictions, allergies) linked by `family_id`
- **`recipes.is_public`** (BOOLEAN) — all-or-nothing public flag, but no UI exposes it
- **`profiles.household_size`** (INT) — informational only
- **RLS on recipes** — `auth.uid() = user_id OR is_public = true`
- **FamilyMembers.tsx** — UI for managing non-auth family profiles (dietary needs, preferences)

### What's Missing
- No way for two authenticated users to share a `family_id` (each signup creates a unique one)
- No invite/accept flow for adding real users to a household
- No middle-ground sharing between "only me" and "everyone"
- No recipe collections/folders
- No per-recipe or per-collection sharing controls
- `family_members` table only tracks non-auth members (kids, etc.) — not linked to `auth.users`

### Implementation Notes (from evaluation, 2026-03-12)
- **`family_members.family_id` uses trigger validation, not a real FK constraint.** The `validate_family_member_family_id` trigger checks that `family_id` exists in `profiles`, but there is no formal `REFERENCES` constraint. Migration step 4 (adding `household_id`) cannot rely on re-pointing a FK — it must add a new column with a real FK to `households(id)` and update the validation trigger accordingly.
- **`handle_new_user()` trigger must be modified.** The existing trigger in migration 002 (`on_auth_user_created`) creates the profile with an auto-generated `family_id`. Post-migration, this trigger must also create a `households` row and insert the new user into `household_members` as `owner`.
- **Existing `family_member` role in `roles` table.** A seeded role `family_member` exists with permissions `{"recipes": true, "view_meal_plans": true}`. The new `household_members.role` (owner/admin/member) is a separate concept from app-level RBAC (`user_roles`). These two systems should remain independent: `user_roles` controls app-level feature access, `household_members.role` controls household-level permissions. Document this distinction clearly.
- **Recipe RLS does not account for collection-level visibility inheritance.** The proposed recipe SELECT policy only checks `recipes.visibility`. If collection sharing inheritance (Area 3) is implemented, the recipe SELECT policy needs an additional OR clause checking `collection_recipes` + `recipe_collections.visibility`. This adds query complexity. Recommendation: defer collection-aware RLS to P2 and document it as a known gap in P0/P1.
- **`family_members` RLS policies need migration.** Current RLS on `family_members` is scoped to `profiles.family_id`. When migrating to `households`, these policies must be rewritten to check `household_members` membership instead.
- **Invite notifications likely need an edge function.** The invite flow (steps 3–4) describes email notifications and auto-join on signup. This requires either a new Supabase Edge Function or a database webhook — neither is currently scoped in Files Affected.

## Scope of Work

### Area 1: Household Management

The household is the core sharing unit. A household is a group of authenticated users who share recipes, meal plans, and preferences.

#### Two Member Types

A household contains two distinct types of members:

| Type | Auth? | How added | Data stored | Use cases |
|------|-------|-----------|-------------|-----------|
| **Authenticated member** (`household_members`) | Yes — has `auth.users` + `profiles` row | Invite flow (email → accept) | Role | Full app access: create/edit recipes, meal plans, chat |
| **Dependent** (`family_members`) | No — no login, no account | Added directly by any household adult | Name, relationship, age (optional), dietary restrictions, allergies, preferences | Tracked for meal planning, future ratings, "who likes this" |

**Key design principles:**
- Dependents (kids, elderly family members, etc.) are managed by authenticated adults — no invite needed, no email collected, no auth credentials
- Dependents already carry dietary restrictions and allergies on the existing `family_members` table. Adding dietary data to `profiles` for authenticated users is a separate concern — see future MOP for dietary/allergy profile enhancements.
- Age on dependents is optional — useful for future portion/nutrition features but not required
- A `managed_by` column on `family_members` tracks which authenticated adult created/manages the dependent (accountability + least privilege)
- Dependents have no app access — they cannot log in, create recipes, or interact with the system directly

#### New Tables

**`households`** — Replaces the implicit `profiles.family_id` grouping

```sql
CREATE TABLE households (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL DEFAULT 'My Household',
  created_by UUID NOT NULL REFERENCES profiles(id),
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

**`household_members`** — Links authenticated users to households with roles

```sql
CREATE TABLE household_members (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  household_id UUID NOT NULL REFERENCES households(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  role TEXT NOT NULL DEFAULT 'member' CHECK (role IN ('owner', 'admin', 'member')),
  joined_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE (household_id, user_id)
);
```

**`household_invites`** — Pending invitations (authenticated members only)

```sql
CREATE TABLE household_invites (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  household_id UUID NOT NULL REFERENCES households(id) ON DELETE CASCADE,
  invited_by UUID NOT NULL REFERENCES profiles(id),
  invited_email TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'accepted', 'declined', 'expired')),
  created_at TIMESTAMPTZ DEFAULT now(),
  expires_at TIMESTAMPTZ DEFAULT now() + INTERVAL '7 days'
);
```

#### Schema Changes to `family_members`

Add `managed_by` and `household_id` to link dependents to the new household model:

```sql
ALTER TABLE family_members
  ADD COLUMN household_id UUID REFERENCES households(id) ON DELETE CASCADE,
  ADD COLUMN managed_by UUID REFERENCES profiles(id);
```

> **Out of scope:** Adding `dietary_restrictions` and `allergies` to `profiles` for authenticated users. Currently only `family_members` tracks dietary data. Extending this to authenticated users (and enabling household-wide dietary aggregation across both member types) should be scoped in a separate MOP.

#### Migration Strategy
1. Create `households` and `household_members` tables
2. For each existing user: create a household, insert the user as `owner`
3. **Update `handle_new_user()` trigger** to also create a household and insert the user as owner on signup
4. Add `household_id` and `managed_by` columns to `family_members` (nullable initially, then backfill by joining `family_members.family_id` → `profiles.family_id` → new `households`; set `managed_by` to the profile that owns the matching `family_id`)
6. Update `validate_family_member_family_id` trigger to validate against `household_id` instead of (or in addition to) `family_id`
7. **Rewrite `family_members` RLS policies** to check `household_members` membership instead of `profiles.family_id`
8. Eventually deprecate `profiles.family_id` (keep for backward compat during transition)

#### Household Roles

| Role | Can invite | Can remove members | Can add dependents | Can delete household | Can manage collections |
|------|-----------|-------------------|-------------------|---------------------|----------------------|
| owner | Yes | Yes | Yes | Yes | Yes |
| admin | Yes | Members only | Yes | No | Yes |
| member | No | No | Yes | No | Own collections only |

> All authenticated household members can add/manage dependents. Dependents do not need invites.

#### Invite Flow (Authenticated Members Only)
1. Owner/admin enters email address
2. System creates `household_invites` row
3. If invitee already has an account → in-app notification + email
4. If invitee doesn't have an account → email with signup link + auto-join on signup
5. Invitee accepts → `household_members` row created, invite marked `accepted`
6. Invites expire after 7 days

#### Adding Dependents (No Invite Required)
1. Any authenticated household member navigates to household settings
2. Fills in: name (required), relationship (optional), age (optional), dietary restrictions, allergies
3. System creates `family_members` row with `household_id` and `managed_by` set to the current user
4. No email sent, no account created, no auth credentials involved

### Area 2: Recipe Permission Model

Three-tier visibility for every recipe:

| Permission | Who Can See | Icon/Label |
|------------|-------------|------------|
| `private` | Only the owner | Lock icon / "Only Me" |
| `household` | Owner + all members of owner's household(s) | Home icon / "My Household" |
| `public` | Anyone (authenticated or not) | Globe icon / "Public" |

#### Schema Change

Replace `recipes.is_public` (BOOLEAN) with `recipes.visibility`:

```sql
ALTER TABLE recipes ADD COLUMN visibility TEXT NOT NULL DEFAULT 'private'
  CHECK (visibility IN ('private', 'household', 'public'));

-- Backfill from existing is_public
UPDATE recipes SET visibility = CASE WHEN is_public THEN 'public' ELSE 'private' END;

-- Eventually drop is_public (after frontend migration)
```

#### Updated RLS Policies

> **Note:** The existing recipe RLS uses two separate SELECT policies (`view own` and `view public`). Both must be dropped and replaced with a single unified SELECT policy.

```sql
-- Drop existing separate SELECT policies
DROP POLICY IF EXISTS "Users can view own recipes" ON recipes;
DROP POLICY IF EXISTS "Users can view public recipes" ON recipes;

-- SELECT: user owns it, OR it's household-shared and user is in the same household, OR public
CREATE POLICY recipes_select ON recipes FOR SELECT USING (
  auth.uid() = user_id
  OR (visibility = 'household' AND EXISTS (
    SELECT 1 FROM household_members hm1
    JOIN household_members hm2 ON hm1.household_id = hm2.household_id
    WHERE hm1.user_id = recipes.user_id
    AND hm2.user_id = auth.uid()
  ))
  OR visibility = 'public'
);

-- INSERT/UPDATE/DELETE: owner only (no change)
CREATE POLICY recipes_insert ON recipes FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY recipes_update ON recipes FOR UPDATE USING (auth.uid() = user_id);
CREATE POLICY recipes_delete ON recipes FOR DELETE USING (auth.uid() = user_id);
```

> **P2 gap:** This policy does not account for collection-level visibility inheritance. When a recipe belongs to a shared collection, a third OR clause will be needed:
> ```sql
> OR EXISTS (
>   SELECT 1 FROM collection_recipes cr
>   JOIN recipe_collections rc ON cr.collection_id = rc.id
>   WHERE cr.recipe_id = recipes.id
>   AND (
>     rc.visibility = 'public'
>     OR (rc.visibility = 'household' AND EXISTS (
>       SELECT 1 FROM household_members hm1
>       JOIN household_members hm2 ON hm1.household_id = hm2.household_id
>       WHERE hm1.user_id = rc.user_id AND hm2.user_id = auth.uid()
>     ))
>   )
> )
> ```
> This is deferred to P2 (Collection-level sharing inheritance) to avoid premature query complexity.

#### UI: Visibility Picker
- Dropdown or segmented control on recipe save/edit
- Three options with icons: Only Me (lock), My Household (home), Public (globe)
- Default: `private`
- Batch update: ability to change visibility for multiple recipes at once

### Area 3: Recipe Collections (Folders)

Collections group recipes and carry their own sharing permissions. Recipes can belong to multiple collections.

#### New Tables

**`recipe_collections`** — Folders for organizing recipes

```sql
CREATE TABLE recipe_collections (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  description TEXT,
  visibility TEXT NOT NULL DEFAULT 'private'
    CHECK (visibility IN ('private', 'household', 'public')),
  icon TEXT,           -- emoji or icon name for UI
  sort_order INT DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE (user_id, LOWER(TRIM(name)))
);
```

**`collection_recipes`** — Many-to-many join

```sql
CREATE TABLE collection_recipes (
  collection_id UUID NOT NULL REFERENCES recipe_collections(id) ON DELETE CASCADE,
  recipe_id UUID NOT NULL REFERENCES recipes(id) ON DELETE CASCADE,
  added_at TIMESTAMPTZ DEFAULT now(),
  sort_order INT DEFAULT 0,
  PRIMARY KEY (collection_id, recipe_id)
);
```

#### Collection Sharing Rules
- A collection's visibility acts as a **floor** for its recipes' effective visibility
- If a collection is `household`, all recipes in it are visible to the household regardless of the recipe's own visibility
- If a collection is `public`, all recipes in it are publicly visible
- A recipe's own visibility still applies outside of collections
- Removing a recipe from a shared collection doesn't change the recipe's own visibility

#### RLS for Collections

```sql
-- Same pattern as recipes: own, household, or public
CREATE POLICY collections_select ON recipe_collections FOR SELECT USING (
  auth.uid() = user_id
  OR (visibility = 'household' AND EXISTS (
    SELECT 1 FROM household_members hm1
    JOIN household_members hm2 ON hm1.household_id = hm2.household_id
    WHERE hm1.user_id = recipe_collections.user_id
    AND hm2.user_id = auth.uid()
  ))
  OR visibility = 'public'
);
```

#### Default Collections
On account creation, auto-create:
- **Favorites** (private) — maps to existing `recipes.is_favorite` flag
- **My Recipes** (private) — all user's recipes (virtual or auto-populated)

#### UI: Collection Management
- Sidebar or tab on Recipes page showing collections
- Drag-and-drop to add recipes to collections
- Collection detail view with grid of recipe cards
- Visibility badge on each collection
- Share button that copies a public link (for public collections)

## Notes

- **Naming decision**: "Household" was chosen over "Family" to be inclusive (roommates, partners, etc.) and to avoid confusion with the existing `family_members` table which stores non-auth profiles.
- **Multi-household**: A user could theoretically belong to multiple households (e.g., both a family household and a cooking club). The schema supports this, but v1 UI will assume one household per user.
- **Ownership stays with creator**: Sharing a recipe to household doesn't transfer ownership. Only the creator can edit/delete. Household members get read-only access. "Fork" (copy to own collection) can be a future feature.
- **`family_members` table**: Continues to exist for tracking non-auth household members (kids, dietary needs). Will be re-linked from `profiles.family_id` to `households.id`.
- **Performance**: The household RLS subquery joins `household_members` twice. For households under ~20 members this is fine. If scaling becomes an issue, a materialized view or denormalized `household_ids[]` array on recipes could help.
- **Two role systems**: `user_roles` (app-level RBAC: admin/user/family_member) and `household_members.role` (household-level: owner/admin/member) are intentionally separate. App roles control feature access; household roles control sharing permissions within a household.
- **Recommended delivery grouping**: Ship P0 (tables + visibility column + RLS) together with the visibility picker UI (currently P1) as a single deliverable. Without the picker, the visibility column has no user-facing value. Invites + collections follow as the next phase.
- **Dietary data on profiles (out of scope)**: Authenticated users currently have no `dietary_restrictions` or `allergies` columns on `profiles` — only dependents in `family_members` track this data. Adding dietary/allergy fields to `profiles` and enabling household-wide dietary aggregation across both member types should be scoped in a separate MOP. This is a prerequisite for features like "filter recipes by household allergies."
- **Future: rating system and "who likes this"**: Both authenticated members and dependents will need to be referenceable as people who rated/liked a recipe. This likely requires a polymorphic reference pattern (e.g., `rated_by_user_id UUID` + `rated_by_member_id UUID`, one nullable) or a unified identity abstraction. Design deferred until the rating feature is scoped.

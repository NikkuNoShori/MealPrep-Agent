## Summary

Users currently own recipes in complete isolation — there's no way to share recipes with family members, a household, or the public in a controlled way. This MOP introduces a three-tier permission model (private / household / public), household management with invite flows, and recipe collections (folders) with inherited sharing permissions.

---

## Current State

### What Exists
- **`profiles.family_id`** (UUID) — groups users into a family, auto-generated on signup (each user gets their own unique `family_id`)
- **`family_members`** table — stores non-authenticated family members (name, relationship, dietary restrictions, allergies) linked by `family_id`
- **`recipes.is_public`** (BOOLEAN) — all-or-nothing public flag, but no UI exposes it
- **`profiles.household_size`** (INT) — informational only
- **RLS on recipes** — `auth.uid() = user_id OR is_public = true`
- **FamilyMembers.tsx** — UI for managing non-auth family profiles (dietary needs, preferences)

### What's Missing
- No way for two authenticated users to share a `family_id` (each signup creates a unique one)
- No invite/accept flow for adding real users to a household
- No middle-ground sharing between "only me" and "everyone"
- No recipe collections/folders
- No per-recipe or per-collection sharing controls
- `family_members` table only tracks non-auth members (kids, etc.) — not linked to `auth.users`

## Scope of Work

### Area 1: Household Management

The household is the core sharing unit. A household is a group of authenticated users who share recipes, meal plans, and preferences.

#### New Tables

**`households`** — Replaces the implicit `profiles.family_id` grouping

```sql
CREATE TABLE households (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL DEFAULT 'My Household',
  created_by UUID NOT NULL REFERENCES profiles(id),
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

**`household_members`** — Links authenticated users to households with roles

```sql
CREATE TABLE household_members (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  household_id UUID NOT NULL REFERENCES households(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  role TEXT NOT NULL DEFAULT 'member' CHECK (role IN ('owner', 'admin', 'member')),
  joined_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE (household_id, user_id)
);
```

**`household_invites`** — Pending invitations

```sql
CREATE TABLE household_invites (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  household_id UUID NOT NULL REFERENCES households(id) ON DELETE CASCADE,
  invited_by UUID NOT NULL REFERENCES profiles(id),
  invited_email TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'accepted', 'declined', 'expired')),
  created_at TIMESTAMPTZ DEFAULT now(),
  expires_at TIMESTAMPTZ DEFAULT now() + INTERVAL '7 days'
);
```

#### Migration Strategy
1. Create `households` and `household_members` tables
2. For each existing user: create a household, insert the user as `owner`
3. Migrate `family_members` rows to reference the new `households.id`
4. Add `household_id` column to `family_members` (nullable initially, then backfill from `profiles.family_id`)
5. Eventually deprecate `profiles.family_id` (keep for backward compat during transition)

#### Household Roles

| Role | Can invite | Can remove members | Can delete household | Can manage collections |
|------|-----------|-------------------|---------------------|----------------------|
| owner | Yes | Yes | Yes | Yes |
| admin | Yes | Members only | No | Yes |
| member | No | No | No | Own collections only |

#### Invite Flow
1. Owner/admin enters email address
2. System creates `household_invites` row
3. If invitee already has an account → in-app notification + email
4. If invitee doesn't have an account → email with signup link + auto-join on signup
5. Invitee accepts → `household_members` row created, invite marked `accepted`
6. Invites expire after 7 days

### Area 2: Recipe Permission Model

Three-tier visibility for every recipe:

| Permission | Who Can See | Icon/Label |
|------------|-------------|------------|
| `private` | Only the owner | Lock icon / "Only Me" |
| `household` | Owner + all members of owner's household(s) | Home icon / "My Household" |
| `public` | Anyone (authenticated or not) | Globe icon / "Public" |

#### Schema Change

Replace `recipes.is_public` (BOOLEAN) with `recipes.visibility`:

```sql
ALTER TABLE recipes ADD COLUMN visibility TEXT NOT NULL DEFAULT 'private'
  CHECK (visibility IN ('private', 'household', 'public'));

-- Backfill from existing is_public
UPDATE recipes SET visibility = CASE WHEN is_public THEN 'public' ELSE 'private' END;

-- Eventually drop is_public (after frontend migration)
```

#### Updated RLS Policies

```sql
-- SELECT: user owns it, OR it's household-shared and user is in the same household, OR public
CREATE POLICY recipes_select ON recipes FOR SELECT USING (
  auth.uid() = user_id
  OR (visibility = 'household' AND EXISTS (
    SELECT 1 FROM household_members hm1
    JOIN household_members hm2 ON hm1.household_id = hm2.household_id
    WHERE hm1.user_id = recipes.user_id
    AND hm2.user_id = auth.uid()
  ))
  OR visibility = 'public'
);

-- INSERT/UPDATE/DELETE: owner only (no change)
CREATE POLICY recipes_insert ON recipes FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY recipes_update ON recipes FOR UPDATE USING (auth.uid() = user_id);
CREATE POLICY recipes_delete ON recipes FOR DELETE USING (auth.uid() = user_id);
```

#### UI: Visibility Picker
- Dropdown or segmented control on recipe save/edit
- Three options with icons: Only Me (lock), My Household (home), Public (globe)
- Default: `private`
- Batch update: ability to change visibility for multiple recipes at once

### Area 3: Recipe Collections (Folders)

Collections group recipes and carry their own sharing permissions. Recipes can belong to multiple collections.

#### New Tables

**`recipe_collections`** — Folders for organizing recipes

```sql
CREATE TABLE recipe_collections (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  description TEXT,
  visibility TEXT NOT NULL DEFAULT 'private'
    CHECK (visibility IN ('private', 'household', 'public')),
  icon TEXT,           -- emoji or icon name for UI
  sort_order INT DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE (user_id, LOWER(TRIM(name)))
);
```

**`collection_recipes`** — Many-to-many join

```sql
CREATE TABLE collection_recipes (
  collection_id UUID NOT NULL REFERENCES recipe_collections(id) ON DELETE CASCADE,
  recipe_id UUID NOT NULL REFERENCES recipes(id) ON DELETE CASCADE,
  added_at TIMESTAMPTZ DEFAULT now(),
  sort_order INT DEFAULT 0,
  PRIMARY KEY (collection_id, recipe_id)
);
```

#### Collection Sharing Rules
- A collection's visibility acts as a **floor** for its recipes' effective visibility
- If a collection is `household`, all recipes in it are visible to the household regardless of the recipe's own visibility
- If a collection is `public`, all recipes in it are publicly visible
- A recipe's own visibility still applies outside of collections
- Removing a recipe from a shared collection doesn't change the recipe's own visibility

#### RLS for Collections

```sql
-- Same pattern as recipes: own, household, or public
CREATE POLICY collections_select ON recipe_collections FOR SELECT USING (
  auth.uid() = user_id
  OR (visibility = 'household' AND EXISTS (
    SELECT 1 FROM household_members hm1
    JOIN household_members hm2 ON hm1.household_id = hm2.household_id
    WHERE hm1.user_id = recipe_collections.user_id
    AND hm2.user_id = auth.uid()
  ))
  OR visibility = 'public'
);
```

#### Default Collections
On account creation, auto-create:
- **Favorites** (private) — maps to existing `recipes.is_favorite` flag
- **My Recipes** (private) — all user's recipes (virtual or auto-populated)

#### UI: Collection Management
- Sidebar or tab on Recipes page showing collections
- Drag-and-drop to add recipes to collections
- Collection detail view with grid of recipe cards
- Visibility badge on each collection
- Share button that copies a public link (for public collections)

## Acceptance Criteria

- [ ] New user signup creates a household with them as owner
- [ ] Owner can invite another user by email
- [ ] Invitee can accept and see owner's `household`-visible recipes
- [ ] Invitee cannot see owner's `private` recipes
- [ ] `public` recipes are visible to all authenticated users
- [ ] Recipe visibility can be changed between private/household/public
- [ ] Collections can be created, renamed, deleted
- [ ] Recipes can be added to / removed from collections
- [ ] Collection visibility controls who can see the collection and its recipes
- [ ] Existing `is_public = true` recipes are migrated to `visibility = 'public'`
- [ ] Existing `is_public = false` recipes are migrated to `visibility = 'private'`
- [ ] All RLS policies pass security audit (no data leakage between households)
- [ ] Family members (non-auth) remain functional after migration

## Notes

- **Naming decision**: "Household" was chosen over "Family" to be inclusive (roommates, partners, etc.) and to avoid confusion with the existing `family_members` table which stores non-auth profiles.
- **Multi-household**: A user could theoretically belong to multiple households (e.g., both a family household and a cooking club). The schema supports this, but v1 UI will assume one household per user.
- **Ownership stays with creator**: Sharing a recipe to household doesn't transfer ownership. Only the creator can edit/delete. Household members get read-only access. "Fork" (copy to own collection) can be a future feature.
- **`family_members` table**: Continues to exist for tracking non-auth household members (kids, dietary needs). Will be re-linked from `profiles.family_id` to `households.id`.
- **Performance**: The household RLS subquery joins `household_members` twice. For households under ~20 members this is fine. If scaling becomes an issue, a materialized view or denormalized `household_ids[]` array on recipes could help.
